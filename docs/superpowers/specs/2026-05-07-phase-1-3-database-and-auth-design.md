# Phase 1.3: Database, Authentication, and Organization Management

## Goal

Set up the database layer with Prisma 7, implement Better Auth with email/password and OAuth providers, enable full organization management, and build the auth UI screens — giving the starter template working multi-tenant authentication out of the box.

## Scope

This phase covers five deliverables:

1. **Infrastructure** — Docker Compose with PostgreSQL, Prisma 7 multi-file schema
2. **Auth engine** — Better Auth in `packages/auth` with email/password, Google/Microsoft OAuth, organization and admin plugins
3. **Auth UI screens** — Sign in, sign up, forgot password, reset password, email verification (via Better Auth UI for shadcn, copied into owned feature components)
4. **Organization UI** — Create org, org switcher, member management, invitation flows (built custom with shadcn + Better Auth org client API)
5. **Route protection** — Middleware, server-side layout checks, client-side reactive protection

### Out of scope

- Two-factor authentication (Phase 2)
- Passkeys (deferred)
- Teams / nested groups within orgs (deferred)
- Admin dashboard UI (Phase 6) — admin plugin is configured and seeded but no admin screens
- Email delivery provider (console-logged links for dev; Resend/NodeMailer deferred)
- Dynamic organization roles (static owner/admin/member only)

### prisma/schema.prisma

Contains the generator and datasource blocks. Prisma 7 requires `output` in the generator:

```prisma
generator client {
  provider = "prisma-client"
  output   = "../src/generated/prisma"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")

```

### prisma/auth.prisma

All Better Auth models: `User`, `Session`, `Account`, `Verification`, `Organization`, `Member`, `Invitation`. These are generated by `npx auth generate` and then owned by us. The schema includes:

- **Core auth:** `User` (id, name, email, emailVerified, image, role, banned, banReason, banExpires, createdAt, updatedAt), `Session` (token, expiresAt, ipAddress, userAgent, userId, activeOrganizationId, impersonatedBy), `Account` (provider, providerAccountId, userId, password, etc.), `Verification` (identifier, value, expiresAt)
- **Organization plugin:** `Organization` (id, name, slug, logo, metadata, createdAt), `Member` (id, organizationId, userId, role, createdAt), `Invitation` (id, organizationId, email, role, status, inviterId, expiresAt)
- **Admin plugin fields:** `User.role` (default "user"), `User.banned`, `User.banReason`, `User.banExpires`, `Session.impersonatedBy`

### PrismaClient singleton (src/client.ts)

```typescript
import { PrismaClient } from "./generated/prisma";

const globalForPrisma = globalThis as unknown as { prisma: PrismaClient | undefined };

export const prisma = globalForPrisma.prisma ?? new PrismaClient();

if (process.env.NODE_ENV !== "production") {
  globalForPrisma.prisma = prisma;
}

export type { PrismaClient } from "./generated/prisma";
```

### Seed script (prisma/seed.ts)

Creates:
- A system admin user (`admin@example.com` / `password123`, role: "admin")
- A sample organization ("Acme Corp", slug: "acme") with the admin as owner
- A regular user (`user@example.com` / `password123`, role: "user") as an org member

The seed uses Better Auth's server API to create users so passwords are properly hashed.

### Key decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Prisma version | 7 | Latest, uses `prisma.config.ts`, `prisma-client` generator |
| Multi-file schema | `prisma/` directory with `schema.prisma` + `auth.prisma` | Clean separation; future phases add `crm.prisma`, `billing.prisma` |
| Client output | `src/generated/prisma` | Required in Prisma 7; import from generated location |
| Seed approach | Better Auth server API | Ensures password hashing matches production behavior |

### Better Auth server config (auth.ts)

```typescript
import { betterAuth } from "better-auth";
import { prismaAdapter } from "better-auth/adapters/prisma";
import { organization, admin } from "better-auth/plugins";
import { prisma } from "@workspace/database";
import { ac, permissions } from "./permissions";

export const auth = betterAuth({
  database: prismaAdapter(prisma, { provider: "postgresql" }),
  experimental: { joins: true },
  emailAndPassword: {
    enabled: true,
    requireEmailVerification: true,
    sendVerificationEmail: async ({ user, url }) => {
      console.log(`[Auth] Verification email for ${user.email}: ${url}`);
    },
    sendResetPasswordEmail: async ({ user, url }) => {
      console.log(`[Auth] Password reset for ${user.email}: ${url}`);
    },
  },
  socialProviders: {
    ...(process.env.GOOGLE_CLIENT_ID && {
      google: {
        clientId: process.env.GOOGLE_CLIENT_ID,
        clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
      },
    }),
    ...(process.env.MICROSOFT_CLIENT_ID && {
      microsoft: {
        clientId: process.env.MICROSOFT_CLIENT_ID,
        clientSecret: process.env.MICROSOFT_CLIENT_SECRET!,
        tenantId: process.env.MICROSOFT_TENANT_ID || "common",
      },
    }),
  },
  plugins: [
    organization({
      ac,
      roles: {
        owner: permissions.owner,
        admin: permissions.admin,
        member: permissions.member,
      },
      sendInvitationEmail: async ({ invitation, inviter, organization }) => {
        console.log(
          `[Auth] Org invite: ${invitation.email} invited to ${organization.name} by ${inviter.name}`
        );
        console.log(`[Auth] Accept URL: /accept-invitation/${invitation.id}`);
      },
    }),
    admin(),
  ],
});
```

### Permissions (permissions.ts)

Static, code-defined permission map using Better Auth access control:

```typescript
import { createAccessControl } from "better-auth/plugins/access";

const statement = {
  organization: ["update", "delete"],
  member: ["create", "update", "delete"],
  invitation: ["create", "cancel"],
  billing: ["manage"],
  apiKey: ["create", "revoke"],
} as const;

export const ac = createAccessControl(statement);

export const permissions = {
  owner: ac.newRole({
    organization: ["update", "delete"],
    member: ["create", "update", "delete"],
    invitation: ["create", "cancel"],
    billing: ["manage"],
    apiKey: ["create", "revoke"],
  }),
  admin: ac.newRole({
    organization: ["update"],
    member: ["create", "update", "delete"],
    invitation: ["create", "cancel"],
    billing: ["manage"],
    apiKey: ["create", "revoke"],
  }),
  member: ac.newRole({
    organization: [],
    member: [],
    invitation: [],
    billing: [],
    apiKey: [],
  }),
};
```

### Guards (guards.ts)

Server-side helpers for server actions and API routes:

- `requireUser()` — reads session from headers, throws 401 if unauthenticated
- `requireSystemAdmin()` — calls `requireUser()`, throws 403 if `user.role !== "admin"`
- `requireOrgPermission(permission: string)` — checks the current user's org membership has the required permission

### Session helpers (session.ts)

- `getCurrentUser()` — returns the current user and session, or null
- `getCurrentOrg()` — returns the active organization and membership for the session, or null

### Org switcher behavior

The `OrgSwitcher` component replaces the current placeholder in `common/ui/org-switcher.tsx`:

- Renders in the sidebar header as a dropdown menu
- Lists all organizations the user belongs to (fetched via `authClient.organization.list()`)
- Current org is highlighted with a checkmark
- Switching calls `authClient.organization.setActive({ organizationId })` and navigates to `/(organization)/[new-org-slug]/`
- "Create Organization" option at the bottom opens `/create-org`
- Shows org logo/avatar + name

### Create organization flow

- Form fields: organization name (required), slug (auto-generated from name, editable, uniqueness checked via `authClient.organization.checkSlug()`), optional logo URL
- On submit: calls `authClient.organization.create()`, which auto-sets the new org as active
- Redirects to `/(organization)/[new-slug]/`

### Onboarding / post-signup routing

After sign-up and email verification:
1. Check if user has any organizations (`authClient.organization.list()`)
2. If none → redirect to `/create-org`
3. If one or more → redirect to the last active org or the first org in the list

This logic lives in the `(dashboard)/layout.tsx` which gates access to the dashboard root.

### Member management

**Member list (`members-page-content.tsx`):**
- DataTable showing member name, email, role, and joined date
- Action column with "Change Role" and "Remove" buttons (permission-gated)

**Invite member (`invite-member-dialog.tsx`):**
- Dialog with email input + role select (admin/member — owners can't be invited, only transferred)
- Calls `authClient.organization.inviteMember({ email, role, organizationId })`
- Invitation email logged to console
- Shows toast confirmation

**Pending invitations (`pending-invitations.tsx`):**

**Update role (`update-member-role-dialog.tsx`):**

**Remove member (`remove-member-dialog.tsx`):**

## 6. Route Protection & Session Management

### Three layers

#### Layer 1: Next.js middleware

`apps/dashboard/middleware.ts` — lightweight session cookie check:

- **Protected routes** (everything except `/sign-in`, `/sign-up`, `/forgot-password`, `/reset-password`, `/verify-email`, `/api/auth/*`): redirect to `/sign-in` if no valid session cookie
- **Auth routes** (`/sign-in`, `/sign-up`): redirect to `/` if already authenticated
- No org membership or permission checks — those are too expensive for middleware

```typescript
export const config = {
  matcher: ["/((?!_next|static|favicon.ico|api/auth).*)"],
};
```

#### Layer 2: Server-side layout checks

- `(organization)/[org-slug]/layout.tsx`: calls `ensureSession` from Better Auth UI + verifies the user is a member of the org matching `[org-slug]` via the auth package's `getCurrentOrg()`. If not a member, redirects to the dashboard root.
- Uses `HydrationBoundary` from React Query to dehydrate session data into the client, so `useSession` calls don't trigger loading states.

#### Layer 3: Client-side reactive protection

- `useAuthenticate` hook from Better Auth UI — subscribes to session state and redirects to sign-in if the session expires or is revoked while the page is mounted
- Used alongside server checks for complete coverage (handles token expiry, sign-out in another tab)

### Session flow

```
Request → Middleware (cookie check) 
       → Layout (server session + org membership check, HydrationBoundary)
       → Page (useAuthenticate for reactive protection)
```
